# Payment 중복 생성 Lock Wait Timeout 재현

## 목적
- `payment(order_id, status)` unique key가 있는 상태에서
- 동일한 `(order_id, status='NOT_STARTED')` 값을 여러 세션이 동시에 `INSERT`할 때
- 어떤 경우에는 `Duplicate entry`가 아니라 `Lock wait timeout exceeded`가 먼저 발생하는 현상을
- k6 없이 SQL 세션만으로 직접 재현한다.

## 전제
- DB: MySQL InnoDB
- 테이블: `payment`
- unique key: `(order_id, status)`
- 예시 상태값: `'NOT_STARTED'`

## 먼저 확인할 것

### 1. 테이블 정의 확인
```sql
SHOW CREATE TABLE payment;
```

여기서 아래와 같은 unique key가 실제로 잡혀 있어야 한다.

```sql
UNIQUE KEY ... (order_id, status)
```

### 2. lock wait timeout 값 확인
```sql
SHOW VARIABLES LIKE 'innodb_lock_wait_timeout';
```

재현을 빠르게 하려면 테스트 세션에서만 짧게 줄여도 된다.

```sql
SET SESSION innodb_lock_wait_timeout = 5;
```

## 재현 1: 가장 단순한 2세션 재현

이 재현은 "먼저 잡은 세션이 커밋하지 않고 오래 들고 있으면, 뒤 세션이 duplicate가 아니라 timeout으로 죽는다"를 보여준다.

### Session A
```sql
SET autocommit = 0;
SET SESSION innodb_lock_wait_timeout = 50;
START TRANSACTION;

INSERT INTO payment (id, amount, order_id, payment_key, status)
VALUES (900000000001, 100.0000, 12345, NULL, 'NOT_STARTED');

-- 여기서 COMMIT 하지 말고 대기
```

### Session B
```sql
SET autocommit = 0;
SET SESSION innodb_lock_wait_timeout = 5;
START TRANSACTION;

INSERT INTO payment (id, amount, order_id, payment_key, status)
VALUES (900000000002, 100.0000, 12345, NULL, 'NOT_STARTED');
```

### 기대 결과
- Session B는 즉시 `Duplicate entry`가 나지 않는다.
- 먼저 Session A가 `(order_id=12345, status='NOT_STARTED')`에 대한 unique index 진입을 선점하고 있으므로,
  Session B는 해당 unique index 엔트리 충돌 확인을 끝내지 못하고 기다린다.
- Session A를 5초 넘게 잡고 있으면 Session B는 아래와 유사한 에러로 종료된다.

```sql
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction
```

### 이 상태에서 의미하는 것
- duplicate key 충돌 자체는 맞다.
- 하지만 Session B는 duplicate 판정까지 가기 전에 락 대기 시간이 먼저 만료된 것이다.

## 재현 2: 같은 SQL에서 duplicate와 timeout이 갈리는 분기 확인

이 재현은 "왜 랜덤처럼 보이는가"를 SQL 세션 수준에서 보여준다.

### Session A
```sql
SET autocommit = 0;
SET SESSION innodb_lock_wait_timeout = 50;
START TRANSACTION;

INSERT INTO payment (id, amount, order_id, payment_key, status)
VALUES (900000000011, 100.0000, 22345, NULL, 'NOT_STARTED');

-- 커밋하지 말고 잠시 대기
```

### Session B
```sql
SET autocommit = 0;
SET SESSION innodb_lock_wait_timeout = 10;
START TRANSACTION;

INSERT INTO payment (id, amount, order_id, payment_key, status)
VALUES (900000000012, 100.0000, 22345, NULL, 'NOT_STARTED');
```

### 분기 A: Session A를 timeout 전에 커밋
Session A에서:
```sql
COMMIT;
```

그러면 Session B는 대기에서 깨어난 뒤 중복을 확인하고 보통 아래로 끝난다.

```sql
ERROR 1062 (23000): Duplicate entry ... for key ...
```

### 분기 B: Session A를 timeout 이후에 커밋
Session A를 10초 넘게 잡고 있으면 Session B는:

```sql
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction
```

즉 같은 충돌 구조라도,
- 먼저 락이 풀리면 `duplicate key`
- 락이 오래 유지되면 `lock wait timeout`
으로 갈린다.

## 재현 3: 3세션으로 대기열 확인

이 재현은 k6처럼 여러 요청이 한 unique key 슬롯에 몰릴 때의 느낌을 더 가깝게 보여준다.

### Session A
```sql
SET autocommit = 0;
START TRANSACTION;

INSERT INTO payment (id, amount, order_id, payment_key, status)
VALUES (900000000021, 100.0000, 32345, NULL, 'NOT_STARTED');

-- 커밋하지 말고 대기
```

### Session B
```sql
SET autocommit = 0;
SET SESSION innodb_lock_wait_timeout = 15;
START TRANSACTION;

INSERT INTO payment (id, amount, order_id, payment_key, status)
VALUES (900000000022, 100.0000, 32345, NULL, 'NOT_STARTED');
```

### Session C
```sql
SET autocommit = 0;
SET SESSION innodb_lock_wait_timeout = 15;
START TRANSACTION;

INSERT INTO payment (id, amount, order_id, payment_key, status)
VALUES (900000000023, 100.0000, 32345, NULL, 'NOT_STARTED');
```

### 기대 결과
- Session B, C 둘 다 대기 상태에 들어갈 수 있다.
- Session A를 빨리 커밋하면 B/C는 duplicate 방향으로 갈 수 있다.
- Session A를 오래 잡고 있으면 B/C 일부 또는 전부가 timeout으로 끝날 수 있다.
- 이것이 k6에서 "어느 요청은 409, 어느 요청은 500"으로 갈리는 원리와 같다.

## 락 상태 직접 확인

MySQL 8.0 이상이면 `performance_schema.data_locks`, `data_lock_waits`로 확인할 수 있다.

### 대기 중 락 보기
```sql
SELECT
    ENGINE_TRANSACTION_ID,
    THREAD_ID,
    OBJECT_SCHEMA,
    OBJECT_NAME,
    INDEX_NAME,
    LOCK_TYPE,
    LOCK_MODE,
    LOCK_STATUS,
    LOCK_DATA
FROM performance_schema.data_locks
WHERE OBJECT_NAME = 'payment'
ORDER BY ENGINE_TRANSACTION_ID, INDEX_NAME, LOCK_DATA;
```

### 어떤 락이 누구를 막는지 보기
```sql
SELECT
    w.REQUESTING_ENGINE_TRANSACTION_ID AS waiting_trx_id,
    w.BLOCKING_ENGINE_TRANSACTION_ID AS blocking_trx_id,
    rl.OBJECT_NAME AS waiting_object,
    rl.INDEX_NAME AS waiting_index,
    rl.LOCK_TYPE AS waiting_lock_type,
    rl.LOCK_MODE AS waiting_lock_mode,
    rl.LOCK_DATA AS waiting_lock_data,
    bl.LOCK_MODE AS blocking_lock_mode,
    bl.LOCK_DATA AS blocking_lock_data
FROM performance_schema.data_lock_waits w
JOIN performance_schema.data_locks rl
  ON w.REQUESTING_ENGINE_LOCK_ID = rl.ENGINE_LOCK_ID
JOIN performance_schema.data_locks bl
  ON w.BLOCKING_ENGINE_LOCK_ID = bl.ENGINE_LOCK_ID
WHERE rl.OBJECT_NAME = 'payment';
```

## 관측 포인트

대기 중일 때 보통 아래 성격의 락이 보인다.

- `TABLE / IX`
  - insert를 수행하는 트랜잭션이 가지는 intention exclusive 테이블 락
- `RECORD / X,INSERT_INTENTION`
  - insert 전에 잡는 insert intention gap lock
- unique index record 관련 대기
  - 같은 `(order_id, status)` 위치에 들어가려는 충돌이 실제로 발생하는 지점

참고로 `LOCK_MODE`의 정확한 문자열은 MySQL 버전과 관측 시점에 따라 조금 다를 수 있다.

## 재현 후 정리

테스트 데이터를 지우려면 마지막에 명시적으로 삭제한다.

```sql
DELETE FROM payment
WHERE order_id IN (12345, 22345, 32345);
```

## 해석
- 이 재현에서 중요한 것은 "같은 unique key에 대한 경쟁이 duplicate key 예외로 곧장 끝나지 않는다"는 점이다.
- 먼저 들어간 트랜잭션이 unique index 충돌 지점을 잡고 있으면 뒤 트랜잭션은 duplicate 판단 전까지 대기한다.
- 그 대기 시간이 `innodb_lock_wait_timeout`을 넘으면 duplicate가 아니라 timeout이 나온다.
