# Analytics 4번째 구현 계획

## Summary

4번째 구현은 `analytics_events` 테이블을 기반으로 KPI 조회 API를 추가한다.

- 범위는 전체 요약 KPI, 상품별 KPI, 검색어별 KPI다.
- 기준은 Funnel API와 동일하게 이벤트 수 기반 `EVENT_COUNT`다.
- 검색어별 CTR 연결을 위해 `PRODUCT_VIEW` 이벤트에 `searchEventId`, `searchKeyword`를 추가한다.
- 상품별 구매율은 현재 이벤트/주문 모델로 정확히 귀속할 수 없으므로 V1에서는 계산하지 않는다.
- Superset, 세그먼트, LLM 인사이트, 카테고리별 KPI는 이번 범위에서 제외한다.

## Design Intent

### KPI 범위

Funnel API 다음 단계는 전체 흐름의 전환율을 더 직접적인 KPI로 제공하는 것이다.

이번 범위는 아래 3가지로 제한한다.

- 전체 요약 KPI
- 상품별 KPI
- 검색어별 KPI

이 범위가 적절한 이유는 현재 `analytics_events`에 저장된 이벤트만으로 대부분의 값을 계산할 수 있고, Funnel에서 사용한 기간 조건과 이벤트 수 기반 계산 방식을 그대로 확장할 수 있기 때문이다.

카테고리별 KPI는 product table과의 join이 필요하고, Superset/세그먼트/LLM은 KPI API 이후에 붙이는 것이 자연스럽다. 따라서 이번 단계에서는 분석 API 자체를 먼저 완성한다.

### 검색어별 CTR 연결

현재 `SEARCH` 이벤트는 `keyword`를 가지고 있지만, `PRODUCT_VIEW` 이벤트에는 어떤 검색에서 이어진 조회인지 알 수 있는 정보가 없다. 이 상태에서는 검색어별 CTR을 정확히 계산할 수 없다.

따라서 `PRODUCT_VIEW` 이벤트에 아래 2개 필드를 모두 추가한다.

- `searchEventId`: 검색 이벤트의 `eventId`
- `searchKeyword`: 상품 조회로 이어진 검색어

`searchEventId`는 특정 사용자의 검색 후 조회 행동을 추적하기 위한 연결 키다. 예를 들어 한 사용자가 어떤 검색 이벤트 이후 어떤 상품을 봤는지 분석할 때 유용하다.

`searchKeyword`는 전체 사용자 기준의 검색어 통계를 빠르게 계산하기 위한 분석 차원이다. 검색어별 CTR, 검색어별 조회 전환 수처럼 대부분의 사용자 행동을 keyword 단위로 집계할 때 self join 없이 사용할 수 있다.

두 값을 함께 저장하는 이유는 목적이 다르기 때문이다. `searchEventId`는 사용자 행동 추적의 정확도를 높이고, `searchKeyword`는 대규모 통계 집계를 단순하게 만든다.

수집 시점에는 `searchEventId`와 `searchKeyword`의 실제 일치 여부를 DB로 검증하지 않는다. 이벤트 수집 API는 빠른 수집을 우선하고, 정합성 평가는 분석 단계나 후속 데이터 품질 작업에서 처리한다.

### 상품별 구매율 제외

상품별 KPI 응답에는 `purchaseRate` 필드를 포함하되, V1에서는 `null`로 반환한다.

현재 `PURCHASE` 이벤트는 `orderId`만 가지고 있고 `productId`가 없다. 또한 현재 주문 모델에도 주문 상품 라인이 없어서 `orderId`로 상품을 역추적할 수 없다.

이 상태에서 상품별 구매율을 계산하면 임의 추정이 되므로, 이번 구현에서는 조회와 장바구니까지의 상품별 KPI만 정확히 제공한다. 상품별 구매율은 `PURCHASE` 이벤트에 상품 정보를 추가하거나 주문 상품 라인을 모델링한 뒤 후속 구현에서 계산한다.

### API 분리

KPI API는 하나의 `dimension` 파라미터로 합치지 않고 아래처럼 분리한다.

- `/api/analytics/kpis/summary`
- `/api/analytics/kpis/products`
- `/api/analytics/kpis/keywords`

전체 요약, 상품별, 검색어별 KPI는 응답 형태와 조회 쿼리가 다르다. 엔드포인트를 분리하면 Controller/Service/DTO가 명확해지고, 각 API의 정렬/limit/쿼리 최적화를 독립적으로 다룰 수 있다.

### 이벤트 수 기반 계산

이번 KPI는 사용자 deduplication 없이 이벤트 수 기준으로 계산한다.

Funnel API가 이미 `EVENT_COUNT`를 기준으로 제공되고 있으므로 KPI도 같은 기준을 사용한다. 이렇게 해야 Funnel과 KPI의 숫자가 서로 설명 가능하고, 추후 사용자 수 기반 KPI를 별도 기준으로 확장하기 쉽다.

## API Contract

### Product View Event 확장

`POST /api/analytics/events/product-view`

```json
{
  "userId": 1,
  "productId": 100,
  "searchEventId": 123,
  "searchKeyword": "macbook"
}
```

- `userId`: 필수, 양수
- `productId`: 필수, 양수
- `searchEventId`: 선택, 값이 있으면 양수
- `searchKeyword`: 선택, 값이 있으면 blank 불가

기존 요청도 계속 허용한다.

```json
{
  "userId": 1,
  "productId": 100
}
```

Kafka 메시지와 저장 이벤트에는 `searchEventId`와 `searchKeyword`를 nullable 값으로 포함한다. 저장 테이블에서는 기존 `keyword` 컬럼을 `SEARCH.keyword`와 `PRODUCT_VIEW.searchKeyword` 저장에 함께 사용하고, `search_event_id` 컬럼을 새로 추가한다.

### Common KPI Rules

- 기간 조건: `from`, `to` 필수
- 시간 형식: ISO-8601 Instant 문자열
- 조회 조건: `occurred_at >= from AND occurred_at < to`
- `from >= to`이면 `400 Bad Request`
- `limit` 기본값은 `20`
- `limit` 허용 범위는 `1..100`
- 분모가 `0`이면 rate는 `0.0`
- 기준은 `EVENT_COUNT`

### Summary KPI

`GET /api/analytics/kpis/summary?from=2026-05-01T00:00:00Z&to=2026-06-01T00:00:00Z`

응답:

```json
{
  "from": "2026-05-01T00:00:00Z",
  "to": "2026-06-01T00:00:00Z",
  "basis": "EVENT_COUNT",
  "searchCount": 100,
  "productViewCount": 40,
  "addToCartCount": 12,
  "purchaseCount": 3,
  "searchCtr": 0.4,
  "cartRate": 0.3,
  "purchaseRate": 0.075
}
```

계산 규칙:

- `searchCtr = productViewCount / searchCount`
- `cartRate = addToCartCount / productViewCount`
- `purchaseRate = purchaseCount / productViewCount`

### Product KPI

`GET /api/analytics/kpis/products?from=2026-05-01T00:00:00Z&to=2026-06-01T00:00:00Z&limit=20`

응답:

```json
{
  "from": "2026-05-01T00:00:00Z",
  "to": "2026-06-01T00:00:00Z",
  "basis": "EVENT_COUNT",
  "items": [
    {
      "productId": 100,
      "productViewCount": 40,
      "addToCartCount": 12,
      "cartRate": 0.3,
      "purchaseRate": null
    }
  ]
}
```

계산 규칙:

- `productViewCount`: `PRODUCT_VIEW` 이벤트를 `productId` 기준으로 집계
- `addToCartCount`: `ADD_TO_CART` 이벤트를 `productId` 기준으로 집계
- `cartRate = addToCartCount / productViewCount`
- `purchaseRate`: V1에서는 `null`

정렬:

- `productViewCount DESC`
- 동률이면 `productId ASC`

### Keyword KPI

`GET /api/analytics/kpis/keywords?from=2026-05-01T00:00:00Z&to=2026-06-01T00:00:00Z&limit=20`

응답:

```json
{
  "from": "2026-05-01T00:00:00Z",
  "to": "2026-06-01T00:00:00Z",
  "basis": "EVENT_COUNT",
  "items": [
    {
      "keyword": "macbook",
      "searchCount": 100,
      "productViewCount": 30,
      "searchCtr": 0.3
    }
  ]
}
```

계산 규칙:

- `searchCount`: `SEARCH.keyword` 기준으로 검색 이벤트 수 집계
- `productViewCount`: `PRODUCT_VIEW.keyword` 기준으로 상품 조회 이벤트 수 집계
- `searchCtr = productViewCount / searchCount`
- `PRODUCT_VIEW.keyword`가 null인 이벤트는 keyword KPI의 `productViewCount`에 포함하지 않는다.

정렬:

- `searchCount DESC`
- 동률이면 `keyword ASC`

`searchEventId`는 keyword KPI의 기본 집계에는 사용하지 않는다. 대신 사용자 단위 검색 후 조회 행동 분석이나 후속 상세 분석에서 `PRODUCT_VIEW.searchEventId -> SEARCH.eventId` 연결 키로 사용한다.

## Implementation Changes

- `ProductViewEventRequest`에 nullable `searchEventId`, `searchKeyword`를 추가한다.
- `AnalyticsEventMessage`에 nullable `searchEventId`를 추가한다.
- `AnalyticsEventMessage.productView(...)`가 `searchEventId`, `searchKeyword`를 받을 수 있게 확장한다.
- `AnalyticsStoredEvent`에 nullable `searchEventId`를 추가한다.
- `AnalyticsStoredEvent.from(...)`은 `PRODUCT_VIEW.searchKeyword`를 기존 `keyword` 컬럼에 저장한다.
- `AnalyticsKpiController`, `AnalyticsKpiService`, KPI 응답 DTO를 추가한다.
- `AnalyticsEventRepository`에 summary/product/keyword KPI 집계용 projection query를 추가한다.
- 운영 DDL은 문서에 남긴다. 현재 프로젝트는 Flyway/Liquibase를 쓰지 않으므로 prod DB에는 별도 적용이 필요하다.

운영 DDL 방향:

```sql
ALTER TABLE analytics_events
    ADD COLUMN search_event_id BIGINT;

CREATE INDEX idx_analytics_events_search_event_id
    ON analytics_events (search_event_id);

CREATE INDEX idx_analytics_events_type_product_occurred_at
    ON analytics_events (event_type, product_id, occurred_at);

CREATE INDEX idx_analytics_events_type_keyword_occurred_at
    ON analytics_events (event_type, keyword, occurred_at);
```

## Test Plan

### Controller Test

- `GET /api/analytics/kpis/summary` 정상 요청 시 `200 OK`와 summary KPI 응답을 반환해야 한다.
- `GET /api/analytics/kpis/products` 정상 요청 시 `200 OK`와 product KPI 응답을 반환해야 한다.
- `GET /api/analytics/kpis/keywords` 정상 요청 시 `200 OK`와 keyword KPI 응답을 반환해야 한다.
- `from`, `to` 누락 또는 잘못된 형식 요청은 `400 Bad Request`를 반환해야 한다.
- `from >= to` 요청은 `400 Bad Request`를 반환해야 한다.
- `limit < 1` 또는 `limit > 100` 요청은 `400 Bad Request`를 반환해야 한다.
- product-view 요청에서 `searchEventId`는 없거나 양수이면 허용해야 한다.
- product-view 요청에서 `searchEventId`가 0 이하이면 `400 Bad Request`를 반환해야 한다.
- product-view 요청에서 `searchKeyword`는 없거나 non-blank이면 허용해야 한다.
- product-view 요청에서 `searchKeyword`가 blank이면 `400 Bad Request`를 반환해야 한다.

### Service Test

- Summary KPI의 `searchCtr`, `cartRate`, `purchaseRate`를 계산해야 한다.
- 각 rate의 분모가 `0`이면 `0.0`으로 처리해야 한다.
- Product KPI의 `cartRate`를 계산해야 한다.
- Product KPI의 `purchaseRate`는 V1에서 `null`이어야 한다.
- Keyword KPI는 `SEARCH.keyword`와 `PRODUCT_VIEW.keyword` 기준으로 `searchCtr`를 계산해야 한다.
- Keyword KPI에서 `PRODUCT_VIEW.keyword`가 null인 이벤트는 조회 수에 포함하지 않아야 한다.

### Repository Integration Test

- 모든 KPI 집계는 `occurred_at >= from AND occurred_at < to` 조건을 지켜야 한다.
- Summary KPI는 기간 내 event type별 count를 정확히 집계해야 한다.
- Product KPI는 `productId`별 `PRODUCT_VIEW`, `ADD_TO_CART` count를 정확히 집계해야 한다.
- Product KPI 정렬은 `productViewCount DESC`, `productId ASC` 순서를 지켜야 한다.
- Keyword KPI는 keyword별 `SEARCH`, `PRODUCT_VIEW` count를 정확히 집계해야 한다.
- Keyword KPI 정렬은 `searchCount DESC`, `keyword ASC` 순서를 지켜야 한다.
- `searchEventId`가 저장되는지 검증해야 한다.

## Assumptions

- KPI는 이벤트 수 기반이며 사용자 수 기반 deduplication은 하지 않는다.
- 기간 기준은 `occurred_at`이며 `created_at`은 조회 조건에 쓰지 않는다.
- 모든 시간 입력은 UTC Instant 문자열을 권장한다.
- `searchEventId`는 사용자별 행동 추적과 후속 상세 분석을 위한 연결 키다.
- `searchKeyword`는 전체 사용자 기준 검색어 통계를 위한 집계 차원이다.
- 수집 시점에는 `searchEventId`와 `searchKeyword`의 실제 일치 여부를 검증하지 않는다.
- 기존 product-view 이벤트는 `searchEventId`, `keyword`가 null일 수 있으므로 keyword KPI의 조회 수 집계에서 제외될 수 있다.
- 상품별 구매율은 구매 이벤트 또는 주문 상품 모델이 확장된 뒤 후속 구현에서 계산한다.
