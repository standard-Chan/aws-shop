# Analytics 2번째 구현 계획

## Summary

2번째 구현은 `Kafka Consumer -> analytics_events 저장`까지 완료한다.

- 퍼널/KPI 조회 API, Superset, DLT 구현, Flyway 도입은 이번 범위에서 제외한다.
- Analytics DB는 같은 datasource 안의 전용 테이블로 논리 분리한다.
- 중복 이벤트는 `eventId` 기준으로 정상 skip한다.

## Implementation Changes

- `analytics_events` 테이블을 추가하고 `event_id`를 PK로 둔다.
- 저장 컬럼은 `event_id`, `event_type`, `user_id`, `occurred_at`, `keyword`, `product_id`, `order_id`, `created_at`으로 고정한다.
- `AnalyticsEventStoreService`가 `AnalyticsEventMessage`를 저장 엔티티로 변환한다.
- `createdAt`은 Consumer 저장 시점에 `Clock`으로 생성한다.
- `search-events`, `product-view-events`, `cart-events`, `purchase-events`를 각각 `@KafkaListener`로 구독한다.
- Consumer group 기본값은 `analytics-event-consumer-group`이다.
- producer가 type header를 보내지 않으므로 consumer JSON default type은 `AnalyticsEventMessage`로 명시한다.
- 현재 프로젝트에 마이그레이션 도구가 없으므로 MySQL DDL은 문서에 남긴다.

## Test Plan

- 저장 서비스 단위 테스트로 신규 저장과 중복 skip을 검증한다.
- Repository 통합 테스트로 H2에서 `analytics_events` 저장/조회, enum 문자열, 시간 필드 보존을 검증한다.
- Consumer 단위 테스트로 listener가 저장 서비스에 메시지를 그대로 위임하는지 검증한다.
- 최종 검증은 `./gradlew test`로 한다.

## Assumptions

- 이번 구현은 “분석 가능한 원천 이벤트를 DB에 남기는 것”이 목표이며 조회 API는 만들지 않는다.
- DLT와 실패 메시지 재처리 정책은 후속 작업으로 남긴다.
- Flyway 도입은 별도 인프라성 작업으로 분리한다.
