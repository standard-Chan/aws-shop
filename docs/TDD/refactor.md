## 리팩터링 규칙
- 테스트 코드는 절대 수정하지 마세요
- 테스트는 계속 GREEN을 유지해야 합니다
- 한 번에 하나씩만 변경 (여러 개 동시에 바꾸지 마)
- 동작(behavior)은 절대 바꾸지 마
- 확장성을 위한 구조 추가 금지 (요청하지 않은 인터페이스/추상화 금지)

## 리팩토링할 사항들
- 절차형 코드 작성 금지
- Service에서 상태 변경 직접 하지 말 것
- Entity는 자기 상태 변경 책임을 가져야 한다
- if 분기 최소화할 것
- 메서드 이름만 보고 의도가 드러나게 작성할 것
- Service는 흐름 제어만 담당하고, 비즈니스 로직은 Entity에 위임할 것
- 생성과 변경 로직을 Entity 내부 메서드로 분리할 것
- 예외 메시지 하드코딩 제거할 것 (상수로 관리)

## 과정
1. 코드에서 냄새나는 부분을 목록으로 알려주세요. 코드는 절대 수정하지 않고, refactor.md 문서로 정리하세요. 
예) 하드코딩된 값, 중복 로직, 긴 메서드, 불명확한 변수명, 긴 조건문, 분리할 수 있는 메서드 등
2. 나의 확인 후, 명시적으로 바꿀 부분만 요청
3. 변경
4. 변경한 내용 및 의도를 ADR 문서로 정리하세요.
5. 테스트가 여전히 GREEN인지 확인 GREEN이 아니라면, 변경한 부분과 실패 원인을 분석해서 다시 수정하세요. 테스트가 GREEN이 될 때까지 반복합니다.

## 리팩토링 참고사항
- Exception을 던질 때는 항상 명확한 메시지를 포함하세요. 앞에 '[]'를 써, 해당 에러를 파악할 수 있게 하고, 그 뒤에 예외에 대한 부연설명을 작성하세요.
- 예) "[상품 생성 실패]: parentAsin이 누락되었습니다."
